# 3.0.0-rc.1 Release Candidate Notes

## Status

This is a breaking release candidate, not a stable release. The candidate is
prepared and verified locally from `develop`; it has not been published to npm,
assigned an npm dist-tag, tagged in Git, pushed, or made available as a GitHub
release. These notes do not announce registry availability.

## Breaking changes

- The package root is Core-only. The removed monolithic `ImageEditor` facade,
  its default export, and Feature method forwarding are not compatibility APIs.
- Applications pass the Fabric module explicitly and install Plugins before
  `init()`, or select a typed Preset.
- Feature options and callbacks move to the Core/Plugin that owns them. Framework
  event handlers call typed Plugin APIs directly; DOM Controls is optional.
- Core accepts `image-editor.state@3`. Recognizable older Snapshots are rejected
  with a migration hint and are never converted implicitly.
- Private controller, manager, source, and `dist` paths are not public entries.
- Fabric `>=7.4.0 <8` is required as a peer; Fabric 8 is not supported by this
  candidate.

See [Migration from 2.x](../migration-from-v2.md) for before/after examples,
method mappings, Snapshot conversion, and the Codemod workflow.

## Core Framework

`ImageEditorCore` owns one Fabric canvas, Base Image lifecycle, layout, image
resource policy, isolated export, typed diagnostics, Snapshot validation, and
Plugin installation. Document, geometry, raster, Overlay, and State work use
coordinated transactions. Validation or execution failure restores prior state;
rollback failure faults the instance instead of reporting a false success.

Core composition is safe to import in SSR/headless modules. Canvas
initialization, image decode, and browser export still need a compatible
Fabric/DOM environment.

## Plugin SDK

The public SDK provides typed `PluginRef<TApi>` identities, validated manifests,
engine/API SemVer ranges, dependency-aware plans, versioned Capabilities,
privileged permissions, disposable setup scopes, lifecycle hooks, Operations,
Tools, State Slices, Mementos, committed events, and shared configuration
contracts. Setup and batch installation are atomic.

The Testing entry exposes profile `3.0` conformance and deterministic hosts for
lifecycle, rollback, dependency, State, Codec, package, bundle, transaction,
permission, and multi-instance proofs. It is isolated from normal runtime
entries.

## Official Plugins

The distribution includes formal entries for Transform, History, the Overlay
Foundation, Mask, Filters, Crop, Mosaic, the Annotation Foundation, Text, Shape,
Draw/Eraser, Overlay State, and DOM Controls. Features expose typed APIs rather
than adding methods to Core. Crop/Mosaic/session previews remain transient;
successful raster and Overlay mutations commit once to the installed History
provider.

## Presets

Minimal, Redaction, Annotation, and Full factories create Core, install a fixed
Plugin plan, and return the editor plus typed Feature APIs. They do not call
`init()`. DOM Controls is absent unless an explicit factory is passed.

## Overlay State

Overlay State wire version 1 remains distinct from the npm package, engine API,
Plugin API, Capability, Snapshot, and State Slice versions. Export and import use
image-normalized coordinates, validate Codecs and resource limits, preserve
ordered persistent objects, exclude transient objects, and commit atomically.

## DOM Controls

The optional DOM adapter binds exact Plugin references to selectors/elements,
status renderers, lists, guarded keyboard actions, and asynchronous error
routing. It reads no browser globals during module evaluation and owns every
listener/subscription it creates.

## Snapshot migration utility

`@bensitu/image-editor/migrate-v2` exports `detectSnapshotVersion`,
`migrateV2Snapshot`, `loadV2Snapshot`, and `v2SnapshotMigration`. Strict mode
rejects unsupported persisted fields instead of silently dropping them. Lossy
conversion requires `unsupportedFieldPolicy: 'warn-and-skip'` and an application
warning sink. Input is bounded and immutable; converted output is revalidated by
Core before any canvas mutation.

Supported frozen data includes Core/Transform state, committed Filters, Masks,
Text/Shape/Draw annotations, and persistent Overlay order/metadata. History
records, selections, active sessions, previews, labels, controls, and other
transient runtime objects are intentionally excluded. Committed Crop/Mosaic
pixels are already represented by the Base Image.

## Codemod

The separately packed `@bensitu/image-editor-codemod` CLI provides the
`v2-to-v3` command. It rewrites common static imports, constructor/options,
Core/Preset composition, ElementMap ownership, and common Transform/History/Mask
calls. Dry-run, diff, write, stable JSON reports, and idempotency are supported.
Dynamic/reflected construction, subclasses, runtime spreads, unsupported
callbacks, mixed versions, and ambiguous DOM maps remain unchanged and are
reported for manual work.

## Package entries

The main package has 22 formal entries covering root/Core, SDK, Testing,
migration, both Foundations, every official Plugin, and all four Presets. Each
entry resolves ESM, CommonJS, ESM declarations, CommonJS declarations, and
NodeNext consumers. The Codemod is a separate package. The complete list and
installation responsibilities are in the [public API reference](../api.md).

## UMD policy

The distribution provides one Full Preset UMD family under `ImageEditorFull`;
CDN metadata selects the minified artifact. Fabric is a separate
global/argument and is not bundled. There is no secondary Core UMD, no
per-Plugin UMD file, no removed-facade alias, and no implicit DOM Controls
installation.

## Dependency updates

The unused `compare-versions` runtime dependency was removed and `tslib` is
build-only. Compatible updates include ESLint 10.7, typescript-eslint 8.64,
globals 17.7, expect-type 1.4, Prettier 3.9.5, Rollup 4.62.2, and Vite 8.1.5.
The final audit must have no unresolved runtime high/critical advisory. Fabric
remains in the supported 7.x peer range.

## Deferred major dependency upgrades

The following majors are review-only and are not adopted here:

- `@rollup/plugin-node-resolve` 16;
- `@rollup/plugin-typescript` 12;
- `@types/jsdom` 28;
- `fast-check` 4;
- `jsdom` 29;
- TypeScript 7;
- Fabric 8.

They require separate compatibility/toolchain work before adoption.

## Known limitations

- Draw Eraser removes whole intersected Draw objects; it does not split paths or
  erase partial path segments.
- The migration utility supports the documented frozen schema inventory, not
  arbitrary application-private fields or runtime objects.
- Plugin permissions are auditable authority boundaries, not a sandbox for
  untrusted JavaScript.
- The Full UMD intentionally includes every official Feature and is larger than
  direct ESM Plugin composition.
- Canvas/image operations are not server-rendered even though public imports and
  types are SSR-safe.

## Upgrade instructions

1. Keep production on the maintained 2.9 line and back up persisted Snapshots.
2. Run the Codemod in `--dry-run` mode on a clean branch and review every
   unresolved finding.
3. Choose a Preset or compose Core + Plugins; move Feature calls/options to the
   owning typed API and dispose Core during framework cleanup.
4. Detect and convert supported older Snapshots through the isolated migration
   entry; do not add migration code to normal startup after stored data is
   converted.
5. Test package entry resolution, strict types, browser editing, rollback,
   History, export, and bundle selection in the application.
6. Consume only an explicitly supplied local candidate tarball for evaluation;
   these notes do not provide an npm installation command because no candidate
   has been published.

The official published 2.9 maintenance baseline is `legacy/v2`.

## Reporting issues

Report candidate issues through the repository
[issue tracker](https://github.com/bensitu/image-editor/issues). Include the
candidate version, runtime/browser, Fabric version, selected entries/Plugins,
minimal reproduction, expected/actual behavior, and whether the failure leaves
State, History, canvas objects, or resources changed.
